# LLM OCR — MinerU batch GUI

WinForms-приложение (.NET 9) для пакетной обработки документов и картинок через MinerU
(PDF, DJVU, DOCX, PPTX, XLSX + изображения: PNG, JPG/JPEG, JP2, WEBP, GIF, BMP, TIFF/TIF)
с двумя режимами вывода:

- **RAG Markdown** — для загрузки в AnythingLLM: HTML-таблицы → markdown pipe-таблицы,
  формулы сохранены как LaTeX-текст, битые OCR-артефакты формул чинятся автоматически.
  По умолчанию ссылки на картинки вырезаются, а весь `.md` валится плоско в выходную папку.
  Галочка **«Сохранять картинки»** переключает на раскладку «папка на файл» (как в HTML):
  каждый документ в свою подпапку `<имя>\`, рядом копируется `images\`, ссылки не вырезаются.
- **HTML с картинками** — самодостаточный HTML с рендером формул (MathJax) и таблиц,
  изображения копируются рядом (`<имя>\<имя>.html` + `images\`).

## Возможности

- Кнопки **Start MinerU / Stop MinerU** и статус-лампа: **зелёная** — сервер `mineru-api`
  жив (`/health`), **красная** — недоступен. Отдельного текстового статуса нет.
- **Авто-перезапуск**: если работавший сервер упал, в панель «Системные сообщения»
  пишется запись и предпринимается попытка поднять его заново.
- Нижняя область лога разделена сплиттером на два пейна (ширины перетаскиваются):
  **слева** — прогресс по файлам, **справа** — системные сообщения (загрузка моделей,
  вывод mineru, события сервера).
- Выбор **бэкенда** (`pipeline` / `hybrid-engine` / `vlm-engine`), **effort**,
  **image-analysis** и **метода парсинга** (`auto` / `txt` / `ocr`) прямо в окне; выбор
  запоминается в `config.json` и восстанавливается при следующем запуске. Бэкенд по
  умолчанию — **hybrid-engine**.
- **Картинки на входе.** Помимо документов принимаются изображения (PNG, JPG, JPEG, JP2,
  WEBP, GIF, BMP, TIFF, TIF) — MinerU читает их напрямую. `.tif` перед обработкой
  переименовывается в `.tiff` (MinerU знает только расширение `tiff`).
- **Ползунок «Держать GPU ≤ N°C» (50–90°C)** — closed-loop термотроттл: подпись показывает
  текущую температуру GPU (через `nvidia-smi`), и пока во время обработки GPU горячее цели,
  дерево процессов mineru приостанавливается (suspend/resume), давая ему остыть; при
  остывании — продолжает. 90°C = без ограничений. Двигается вживую, прав администратора не
  требует. (Тот же подход, что в llmtranslate — cooldown по температуре, адаптирован под
  внешний процесс mineru.)
- **Пропуск готовых (resume пачки):** файл, для которого выходной результат уже есть
  (непустой `.md`/`.html` в ожидаемом месте), пропускается — в логе «уже готово, пропускаю».
  Так пачку можно доганять после сбоя, не переделывая готовое. Чтобы пересобрать файл
  заново — удали его выходной результат.
- При старте обработки, если сервер не поднят, он поднимается автоматически командой
  из `config.json`; вся пачка гонится через `--api-url`.
- DJVU конвертируется в PDF через `ddjvu` (DjVuLibre рядом).
- При выходе из приложения сервер гасится (kill дерева процессов + Job Object).

## Конфигурация (`config.json`, рядом с exe)

| поле | назначение |
|------|-----------|
| `MineruDir` | папка со `mineru.exe` / `mineru-api.exe` (venv Scripts) |
| `Host` / `Port` | адрес сервера mineru-api |
| `Backend` | `hybrid-engine` (по умолч.) / `pipeline` / `vlm-engine` |
| `Effort` | `medium` (по умолч.) / `high` — только для hybrid (см. ниже) |
| `ImageAnalysis` | `auto` / `on` / `off` — анализ картинок/графиков (hybrid и vlm) |
| `Lang` | подсказка языка OCR (напр. `cyrillic`, `east_slavic`); пусто = дефолт |
| `Method` | метод парсинга `auto` / `txt` / `ocr` (`-m`) — только pipeline и hybrid; для vlm игнорируется |
| `ModelSource` | текущее/дефолтное зеркало моделей (значение `MINERU_MODEL_SOURCE`) |
| `ModelSources` | список зеркал для выпадашки в UI (по умолч. `huggingface`, `modelscope`, `local`) — правь здесь, чтобы добавить/убрать |
| `OutputModeName` | запомненный режим вывода: `rag` / `html` |
| `RagSaveImages` | `true`/`false` — раскладка RAG «папка на файл» с картинками |
| `CudaPath` | папка CUDA Toolkit; вписывается как `CUDA_PATH` (+bin в PATH) в env mineru, чтобы GPU-движки (lmdeploy/vLLM) стартовали, даже если родитель не унаследовал машинный `CUDA_PATH`. Пусто/нет папки = пропустить |
| `TargetTempC` | 50–90, целевая макс. температура GPU (термотроттл; 90 = без ограничений) |
| `NvidiaSmiPath` | путь/команда `nvidia-smi` для чтения температуры GPU |
| `DdjvuExe` | путь к `ddjvu.exe` |
| `DdjvuDpi` | DPI при конвертации DJVU→PDF |
| `HealthTimeoutSec` | ожидание готовности сервера |
| `MathJaxUrl` | CDN MathJax для HTML-режима |

Бэкенд, язык, effort, image-analysis, метод, зеркало моделей, режим и галочку картинок можно менять
прямо в окне — любое изменение сразу пишется в `config.json`, так что при следующем старте
восстанавливается последний выбор. Список зеркал в выпадашке берётся из `ModelSources`,
а `ModelSource` задаёт выбранное по умолчанию.

## Backend / Effort / Image-analysis — что когда выбирать

**Бэкенды.** `pipeline` — только классические модели (быстрый, но грубее по формулам и
раскладке; может «склеивать» слова и путать сложные таблицы). `hybrid-engine` (дефолт) —
VLM + pipeline с маршрутизацией по регионам страницы: заметно чище формулы и макет,
на 8 ГБ VRAM влезает через движок `transformers` (lmdeploy требует системный CUDA toolkit).
`vlm-engine` — одна VLM на всю страницу: максимальная точность таблиц и сложных макетов,
медленнее.

**`--effort` (только hybrid).** `medium` — быстро, анализ картинок/графиков **выключен**;
покрывает 90 % задач (текст, формулы, таблицы, код). `high` — включает анализ
картинок/графиков (VLM описывает рисунок текстом) + чуть выше общая точность, но заметно
медленнее и грузит GPU.

**`--image-analysis` (hybrid и vlm).** Превращает содержимое картинок и графиков в
**текстовое описание**, которое попадает в markdown и, значит, индексируется в RAG.
`auto` — не передавать флаг (дефолт MinerU = вкл для vlm и hybrid-high; hybrid-medium его
принудительно выключает). `on` / `off` — явно включить/выключить. Смысл только если нужно,
чтобы содержимое рисунков искалось; для чистого текста — лишняя работа.

**`-m` / метод парсинга (`auto` / `txt` / `ocr`; pipeline и hybrid-*).** Как MinerU достаёт
текст из PDF. `auto` (дефолт) — сам решает: есть текстовый слой → `txt`, нет → `ocr`. `txt` —
брать готовый текстовый слой PDF (быстро и точно, если слой валиден). `ocr` — игнорировать
текстовый слой и распознавать по отрисованным пикселям (нужно для PDF с битыми/подменёнными
шрифтами, где `txt` даёт кракозябры). Для `vlm-engine` флаг не применяется (выпадашка гаснет);
для картинок на входе метод роли не играет — они и так распознаются как пиксели.

### Сравнение бэкендов по типу материала

Плюсы (+) и минусы (−) каждого движка. Коротко: `pipeline` — детерминированный OCR +
экспертные модели (быстрый, **не галлюцинирует**, но грубее и ниже общая точность ~86 %);
`vlm-engine` — одна VLM (семантически чище, точность таблиц ~95 %, но **склонна к
галлюцинациям** на плотных/сложных страницах); `hybrid-engine` — VLM с подстраховкой
pipeline-моделей (баланс, дефолт).

| Материал | `pipeline` | `hybrid-engine` | `vlm-engine` |
|----------|-----------|-----------------|--------------|
| **Сложные формулы** | − путает символы/индексы, грубоватый LaTeX | **+ чистый LaTeX + сверка pipeline (лучший баланс)** | + очень чистый LaTeX; − на тяжёлых страницах уверенная ошибка в символе |
| **Графики (charts/plots)** | − только вырежет как картинку, смысла в текст не даст | + при `high`/`image-analysis` описывает график текстом | + описывает график; − может выдумать значения на осях |
| **Таблицы** | + спец-модель, хорошо со слитыми ячейками, **не выдумывает числа**; − структура грубее | **+ высокая точность (~95 %), баланс** | + макс. структурная точность на обычных; − на плотных может галлюцинировать значения |
| **Картинки-изображения** (фото, схемы) | − только копия файла, без понимания | + `high`/`image-analysis` даёт подпись/описание | + описание содержания; − на декоративных — трата и риск фантазий |
| **Код на разных ЯП** | − OCR склеивает, теряет отступы, путает `l/1/I`, скобки | **+ аккуратнее с блоками кода и отступами** | + лучшая структура; − может «поправить» синтаксис по смыслу |
| **Простой литературный текст** | **+ быстро и надёжно, этого хватает** | + отлично, но избыточно по ресурсам | + отлично; − самый медленный ради нулевой выгоды |
| **Галлюцинации над данными** (формулы/таблицы/картинки/код) | **+ практически нет: ошибки локальные (искажение символа), ничего не выдумывает** | ± умеренный риск: VLM может фантазировать, pipeline-модели подстраховывают | − **наибольший риск**: на сложных страницах выдумывает числа/символы/строки, ошибки «уверенные» и незаметные |

**Про галлюцинации.** Ключевое отличие: pipeline **искажает** (кривой символ, слипшиеся
слова — видно глазом), а VLM может **выдумать** правдоподобное и неверное (не тот числовой
разряд в таблице, «починенный» код, несуществующая подпись к графику) — и это тихо утечёт в
RAG. Поэтому для data-critical материала (числа, формулы, таблицы, код, где один неверный
символ фатален) безопаснее `pipeline` или `hybrid`; чистый `vlm` бери на аккуратных макетах,
где важнее семантическая чистота, и перепроверяй данные.

### Рекомендации по типу материала

| Материал | Backend + настройки | Почему |
|----------|--------------------|--------|
| **Много сложных формул** | hybrid, effort **medium** | формулы распознаёт отдельная модель; VLM даёт чистый LaTeX, image-analysis не нужен. Если формулы «вшиты» в рисунки — effort high |
| **Много графиков (charts/plots)** | hybrid **high** или vlm + **image-analysis on** | единственный способ загнать смысл графика (тренды, оси, значения) в текст и сделать его искомым в RAG |
| **Много таблиц** | hybrid medium; сложные/вложенные — **vlm** | таблицы парсит table-модель независимо от effort; vlm точнее на хитрых макетах. image-analysis для таблиц не нужен |
| **Много картинок-изображений** (фото, схемы) | содержательные — hybrid **high** / vlm **image-analysis on**; декоративные — medium + галочка «сохранять картинки» | описание имеет смысл для схем/диаграмм; для декора — только сохранить файлы для человека, анализ не тратить |
| **Много кода на разных ЯП** | hybrid medium (image-analysis off) | код — это текст; hybrid/vlm бережнее к отступам и блокам, чем pipeline OCR. Код на скринах — тогда image-analysis on / vlm |
| **Простой литературный текст** | pipeline или hybrid medium, image-analysis off | самый лёгкий случай, максимум скорости; high и анализ картинок только замедлят |

**Правило большого пальца:** `hybrid + medium` — рабочий дефолт почти для всего.
Повышай до `high` / `image-analysis on` **только** когда нужно, чтобы содержимое
картинок и графиков искалось в RAG. `vlm-engine` — когда критична точность таблиц и
сложных макетов и не жалко скорости.


## Использование

1. (Опц.) Start MinerU — дождаться зелёной лампы (первый запуск бэкенда докачивает модели;
   в панели «Системные сообщения» видно прогресс загрузки).
2. Добавить файлы (PDF/DJVU/DOCX/PPTX/XLSX или картинки PNG/JPG/JPEG/JP2/WEBP/GIF/BMP/TIFF/TIF)
   и указать выходную папку.
3. Выбрать режим (RAG / HTML), при желании backend, effort, image-analysis, метод и язык.
   Для RAG можно включить «Сохранять картинки».
4. «Обработать пачку». Слева идёт прогресс по файлам, справа — вывод mineru/сервера.
   Результаты:
   - RAG (плоско): `<выход>\<имя>.md` (картинки вырезаны)
   - RAG + «Сохранять картинки»: `<выход>\<имя>\<имя>.md` + `images\`
   - HTML: `<выход>\<имя>\<имя>.html` + `images\`
